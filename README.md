# Jones — Automation Exercise

Playwright automation for the **Junior Automation Engineer** home assignment.

The automation drives the lead form on <https://test.netlify.app/>: it fills in
the contact fields, changes the *Number of Employees* dropdown (bonus), captures
a screenshot before submitting, clicks **Request a call back**, and verifies that
the thank-you page is reached.

## Tech stack

- [Node.js](https://nodejs.org/)
- [@playwright/test](https://playwright.dev/) (Chromium)

## Installation

```bash
npm install
npx playwright install chromium
```

## How to run

```bash
npm test            # headless (default)
npm run test:headed # watch it run in a real browser
npm run test:debug  # step through with the Playwright Inspector
```

When the test reaches the thank-you page it prints:

```
Reached the thank you page successfully.
```

## What the test does

1. Opens <https://test.netlify.app/> and asserts the form is loaded.
2. Fills **Name, Email, Phone, Company, Website** and verifies the values.
3. **Bonus:** changes **Number of Employees** from `1-10` to `51-500`.
4. Saves a screenshot to `screenshots/before-request-call-back.png` *before* submitting.
5. Clicks **Request a call back**.
6. Asserts the URL is `thank-you.html` and the **Thank You!** heading is visible,
   then logs a success message to the console.

## Notes on selector strategy & good practices

- **Semantic locators first.** Every field on the page has a `<label for="...">`,
  so the Page Object uses `getByLabel(...)` and `getByRole('button', ...)` instead
  of brittle CSS/XPath. These mirror how a user perceives the page and survive
  styling changes.
- **Page Object Model.** All page interaction lives in `src/leadFormPage.js`, so
  the spec reads as a clear, business-level flow.
- **Web-first assertions.** `expect(locator).toBeVisible()` / `toHaveValue()` /
  `expect(page).toHaveURL()` auto-wait, so there are **no arbitrary `waitForTimeout`
  calls** in the test.
- **Centralized test data** in `src/testData.js`.
- Debugging artifacts (video / trace / failure screenshot) are produced **only on
  failure** to keep successful runs fast and clean.
